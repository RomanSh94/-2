# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

X20 is a Telegram bot providing AI-assisted emotional support, built on `aiogram 3.7` + OpenAI `gpt-4o-mini` + SQLite (via `aiosqlite`) + a Flask admin dashboard. Bilingual (RU/EN). Target runtime: Python 3.12 or 3.13.

## Commands

Setup and run (PowerShell, Windows):

```powershell
python -m venv venv
.\venv\Scripts\Activate.ps1
pip install --upgrade pip
pip install -r requirements.txt
copy .env.example .env   # then fill BOT_TOKEN, OPENAI_API_KEY, ADMIN_PASSWORD
python bot.py            # starts bot + dashboard at http://localhost:8080
```

There are no tests, linters, or build steps in this repo. The only entry point is `python bot.py`, which calls `init_db()`, starts the Flask dashboard on a background thread, starts the APScheduler check-in job, and enters aiogram polling.

Python version note: Python 3.14 has install issues for some deps; the repo ships parallel files for that case — `language_detector_py314.py`, `notifications_py314.py`, and `requirements_win_py314.txt` / `requirements_py312.txt`. Default code uses the non-suffixed modules; don't import the `_py314` variants unless explicitly switching toolchain.

## Architecture: the pipeline is the whole program

The core design constraint — repeated throughout the code — is **the LLM never decides safety; deterministic code does.** GPT-4o-mini only generates the conversational response inside a tightly scoped scenario; everything that gates *whether* and *how* it speaks is rule-based Python.

[bot.py:pipeline()](bot.py#L61) runs every text message through these stages in order, and each stage corresponds to one module. To make non-trivial changes you almost always need to understand this whole chain:

1. [language_detector.py](language_detector.py) — RU/EN by Cyrillic-vs-Latin char count (no langdetect dep at runtime).
2. [risk_detector.py](risk_detector.py) — pattern-based scoring of 9 categories (suicide, self_harm, panic, hopelessness, dissociation, dependency, loneliness, burnout, aggression). Explicit phrases weighted 1.0, implicit 0.6, with compound-rule bonuses. Returns `score`/`level`/`categories`.
3. **Crisis override** — if `suicide` or `self_harm` is in categories, the pipeline stops, sends the deterministic crisis text from [prompts.py](prompts.py), and notifies admins. No LLM call happens.
4. [state_engine.py](state_engine.py) — rolling per-user emotional state (anxiety, panic, overwhelm, energy, openness, etc.) with decay; persisted as JSON in `user_states.state_json`.
5. [stage_detector.py](stage_detector.py) — `ACUTE_DISTRESS` / `REFLECTION` / `PROBLEM_SOLVING` / `GROWTH` / `OPEN`. **This drives contraindications**: `ACUTE_DISTRESS` blocks CBT and ACT scenarios entirely.
6. [readiness_engine.py](readiness_engine.py) — `LOW` / `MEDIUM` / `HIGH` from explicit phrases.
7. [cognitive_capacity.py](cognitive_capacity.py) — float derived from state; `< 0.3` forces somatic over CBT.
8. [state_engine.py:choose_scenario()](state_engine.py#L57) — picks one of 8 scenarios (`crisis`, `grounding`, `stabilization`, `cbt_thought`, `act_acceptance`, `reflective`, `somatic`, `open_chat`) from state + categories + stage + readiness + capacity + A/B variant.
9. [relationship_monitor.py](relationship_monitor.py) — separate dependency check; if triggered, sends a deterministic redirect and stops the pipeline.
10. [practice_registry.py](practice_registry.py) — selects a **versioned, pre-written structured practice** by category/stage/severity. Practices are not generated by the LLM; the registry is the source of truth.
11. [database.py:log_router_decision()](database.py#L337) — snapshots the entire routing decision for later research analysis.
12. [memory.py](memory.py) — rolling summarization. When unsummarized messages exceed `THRESHOLD=20`, compresses all but the last `KEEP=8` into a stored summary; subsequent prompts include the summary + recent messages only.
13. LLM call — `gpt-4o-mini`, `temperature=0.65`, `max_tokens=300`, system prompt from `prompts.get_system_prompt(scenario, lang)`.
14. [safety_validator.py](safety_validator.py) — runs against every LLM response: forbidden phrases (diagnoses, love declarations, dependency-encouraging text, certainty claims) and a hard `> 150 words` length cap. On failure, the response is replaced with a fallback and the block is logged to `validator_blocks`.
15. Outcome tracking — for non-`crisis` / non-`open_chat` scenarios, the bot prompts for a before-score (1–10) via inline keyboard, then runs the practice, then prompts for after-score, then a 👍/➖/👎 quality rating. Three callback handlers (`cb_before`, `cb_after`, `cb_quality` in bot.py) drive this state machine using **aiogram FSM** (`InterventionStates` + `MemoryStorage`) — not a global dict. The score scale is **1 = bad, 10 = good**, so improvement means `after_score > before_score`.

When adding signals (new risk categories, state dimensions, stage cues, dependency phrases, forbidden output phrases), they must be added in **both** `ru` and `en` lists — every detector loops over both languages regardless of detected language, so missing one side silently weakens detection.

## Database layer

[database.py](database.py) holds the entire schema as a single `SCHEMA` string executed by `init_db()`. There is **no migration system** — schema changes require either dropping `x20.db` or hand-writing ALTER statements. Tables of note: `intervention_results` (the research outcome table — before/after/follow-up scores, `feedback_rating` for 👍/👎, confidence, engagement JSON), `router_decision_logs` (state snapshot at routing time), `adverse_events` (logged when `after_score < before_score - 2`, i.e. significant worsening on 1=bad/10=good scale), `weekly_progress_snapshots`, `validator_blocks`. The `response_quality` table exists in schema but is not written to — quality data lives in `intervention_results.feedback_rating`.

The file mixes two async/sync access styles intentionally:
- **Async (`aiosqlite`)** functions are used by the bot.
- **Sync (`sqlite3`)** functions prefixed `sync_` exist solely for the Flask dashboard, which runs on a separate thread and cannot share the aiosqlite event loop. Don't call `sync_*` from bot code, or async functions from dashboard code.

The `users.message_count` is incremented on every `upsert_user` call (via `ON CONFLICT`), which happens on every inbound message — it counts inbound user messages, not total messages.

## Dashboard

[dashboard.py](dashboard.py) is a single-file Flask app started by `start_dashboard()` on a background thread from `bot.py`. HTML is inlined as f-strings with Bootstrap CSS. Auth is a single password (`ADMIN_PASSWORD`) stored in the session. There are no templates, no static assets, no JS framework — adding a new dashboard page means adding a route in this file.

## A/B testing

[ab_testing.py:get_variant()](ab_testing.py#L15) is `user_id % len(AB_VARIANTS)`. **Deterministic and stateless** — the same user always lands in the same variant; no DB assignment table. The only variant behavior currently wired is in [state_engine.py:choose_scenario()](state_engine.py#L80): `variant_a` picks `act_acceptance` instead of `cbt_thought` for anxious users. Variants are read from `AB_VARIANTS` env var (comma-separated).

## Conventions

- All language-specific data structures (`PATTERNS`, `STATE_SIGNALS`, `STAGE_SIGNALS`, `DEPENDENCY_SIGNALS`, etc.) are keyed by `"ru"` / `"en"` at the inner level; detectors iterate `{lang, "ru", "en"}` so a phrase in *either* language can trigger a hit. Maintain this when adding categories.
- Practice IDs are versioned (`grounding_5senses_v1`); `PRACTICE_VERSION` in [config.py](config.py) is logged on every intervention so research data can distinguish practice revisions. Bump the suffix rather than editing in place if practice text changes meaningfully.
- `ROUTER_VERSION` in [config.py](config.py) is logged with each routing decision for the same reason — bump it when routing logic changes meaningfully.
- Scenario names are string keys used across `state_engine.choose_scenario`, `prompts.PROMPTS`, `practice_registry.CATEGORY_MAP`, and the dashboard outcome stats. Renaming requires touching all four.
- Crisis text, dependency text, and onboarding text live in [prompts.py](prompts.py) as plain strings — these are user-visible and never go through the LLM.
