# MIGRATION_PLAN

План эволюции X20 от текущего рабочего бота к продуктовой версии по 4 эпикам.
Принцип неизменен: **LLM — разговорный слой, safety — детерминированный код.**

Статус базы: текущая боевая БД — SQLite (`x20.db`). НЕ УДАЛЯТЬ.
Миграций нет — схема в одной строке `SCHEMA` в `database.py`.

---

## Порядок и риски (высокоуровнево)

| # | Эпик | Что меняем | Риск | Оценка |
|---|------|-----------|------|--------|
| 1 | Crisis Protocol | crisis_protocol.py, tel:-кнопки, таблица `crisis_events`, follow-up'ы, #CRITICAL алерты | Средний (затрагивает кризисный путь — самый чувствительный) | 2–3 дня |
| 2 | Humanization | вариативность приветствий, голос персоны, анти-робот детектор, typing-задержки, EmotionAnalysis, долгая память (pgvector), GDPR /memory /forget_all | Средний | 4–6 дней |
| 3 | Silence Engine | Celery + Redis, push-тиры с антиспамом | Высокий (новая инфраструктура) | 3–5 дней |
| 4 | Practices Library | 40+ практик, smart-selector по user_need | Низкий | 3–4 дня |
| — | SQLite → PostgreSQL 16 + pgvector | смена СУБД | **Высокий** | отдельный эпик |

## Эпик 1 — Crisis Protocol (детально)

Цель: усилить кризисный путь, не ломая текущую инвариант «при suicide/self_harm LLM не вызывается».

Текущее состояние (проверено в `bot.py`):
- При категориях `suicide`/`self_harm` пайплайн уже останавливается: push_alert админам +
  `message.answer(get_crisis_text(lang))` + `return`. LLM не вызывается. ✅
- `aggression` → только push_alert.

Шаги Эпика 1:
1. **crisis_protocol.py** — выделить уровни RED/ORANGE/YELLOW/GREEN из уже существующего
   risk-скоринга (не дублировать детектор, а маппить его `level`/`categories`).
2. **tel:-инлайн-кнопки** — кнопка прямого звонка на горячую линию (RU: 8-800-2000-122).
   ⚠️ Риск: Telegram inline-кнопки с `url="tel:..."` поддерживаются не на всех клиентах —
   нужен фолбэк (текст номера остаётся в сообщении, как сейчас).
3. **Таблица `crisis_events`** — ALTER/добавление в `SCHEMA` (SQLite, без смены СУБД):
   user_id, ts, level, categories, message_excerpt, admin_notified, resolved.
4. **Колбэки `safe`/`still`** — пользователь подтверждает состояние после кризисного сообщения.
5. **Follow-up'ы 1ч / 24ч / 7д** — через существующий APScheduler (не Celery на этом этапе).
6. **#CRITICAL админ-алерты** — расширить текущий push_alert структурированным сообщением.
7. **tests/test_crisis_protocol.py** — каждое изменение покрыто тестом (правило «фикс = тест»).

### Инженерная рекомендация
**Отложить миграцию SQLite → PostgreSQL (Эпик 1.6) в отдельный эпик.**
Жизне-критичные функции (tel:-кнопка, `crisis_events`, follow-up'ы) должны выйти первыми
на текущей SQLite. Смена СУБД — высокий риск (схема, async-доступ, dashboard sync-доступ,
дев-окружение) и не блокирует ни одну кризисную фичу. pgvector реально нужен только в
Эпике 2 (долгая память), туда его и привязать.

## Эпики 2–4 — границы (детализируются перед стартом каждого)

- **Эпик 2 Humanization**: новые сигналы добавляются в ОБА списка ru/en. pgvector +
  text-embedding-3-small для долгой памяти. GDPR-команды `/memory`, `/forget_all`.
- **Эпик 3 Silence Engine**: Celery + Redis — первая внешняя инфраструктура, требует
  отдельного обсуждения по деплою. Антиспам обязателен до любых push.
- **Эпик 4 Practices Library**: расширение `practice_registry.py`, версионирование ID
  сохраняется (`*_v1`), `PRACTICE_VERSION` бампится при изменении текста.
